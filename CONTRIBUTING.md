# CONTRIBUTING.md

Благодарим за интерес к проекту **DataPipeline Hub**! Мы рады любым улучшениям: от исправления опечаток до новых функциональных модулей.


## Как внести вклад


1. **Создайте issue**  
   Перед началом работы убедитесь, что подобная задача/идея ещё не обсуждается. Создайте issue с меткой:
   - `bug` — для отчётов об ошибках;
   - `feature` — для предложений новых функций;
   - `documentation` — для улучшений документации;
   - `question` — для уточняющих вопросов.

2. **Обсудите решение**  
   В issue опишите:
   - суть предлагаемого изменения;
   - предполагаемый способ реализации;
   - возможные риски/ограничения.

3. **Создайте ветку**  
   ```bash
   git checkout -b feature/your-feature-name
   ```

4. **Внесите изменения**  
   Соблюдайте:
   - стандарты кода проекта (см. раздел «Стиль кода»);
   - принцип атомарных коммитов (один коммит — одно логическое изменение).

5. **Протестируйте**  
   Убедитесь, что:
   - все тесты проходят (`pytest`);
   - новый код покрыт тестами (если применимо);
   - нет критических зависимостей от внешних сервисов.

6. **Создайте Pull Request (PR)**  
   В описании PR укажите:
   - ссылку на issue;
   - краткое описание изменений;
   - инструкции по проверке (если нужно).


## Правила оформления кода

- **Python**: PEP 8, тип‑аннотации обязательны.
- **YAML/JSON**: отступы — 2 пробела, ключи в `snake_case`.
- **SQL**: ключевые слова — заглавные (`SELECT`, `FROM`), имена таблиц/полей — `snake_case`.
- **Имена коммитов**:  
  ```
  <тип>(<область>): <краткое описание>
  ```  
  Примеры:  
  - `fix(api): handle 404 errors`  
  - `feat(dashboard): add user stats`  
  - `docs(readme): update installation guide`

## Типы коммитов

- `feat` — новая функциональность;
- `fix` — исправление ошибок;
- `refactor` — переработка кода без изменения функционала;
- `docs` — изменения документации;
- `test` — добавление/изменение тестов;
- `chore` — рутинные задачи (обновление зависимостей и т.п.).


## Процесс ревью

1. PR проверяется на:
   - соответствие коду стиля;
   - покрытие тестами;
   - наличие документации;
   - отсутствие регресса (CI/CD).
2. Возможны запросы на доработки. Отвечайте на комментарии и вносите изменения в ту же ветку.
3. После одобрения PR мерджится в `main`.


## Что НЕ стоит делать


- Отправлять код без тестов (если это не тривиальное изменение).
- Вносить изменения в продакшен‑конфиги (`config/prod.env`) без согласования.
- Использовать нелицензированные зависимости.
- Нарушать семантическое версионирование (для релизов).
